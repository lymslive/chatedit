# 聊天记录文件格式说明

基本原则：
- 兼容 markdown 格式
- 以 `##` 开头的（二级标题）表示对话内容
- 以 `#` 开头的（一级标题）表示注释行或段落
- 以 `@` 开始的单行表示引入文件
- 以 `!` 开始的单行表示执行一个 shell 命令，捕获标准输出

可提取每段二级标题下的对话内容，转换为发往大模型 API 的一个 message 数组元素。

## 核心对话段

对话段标题的特殊格式是 `## role >>`，其中 role 支持如下几种典型色：
- system : 系统示词，单字母缩写 P
- user : 用户问题，单字母缩写 Q
- assistant : 模型回复，单字母缩写 A

首行 `## role >>` 后面可以有内容，忽略前导空白，故可能有三种基本情况：
- 单行，在 `>>` 标记后直接写上简短内容
- 多行，首行仅标题标记，后续行为正式内容
- 多行，混合前两者，首行有一句话，续行更多详细内容

在多行续行长内容的情况下，允许有次级标题 (`###` 以上) 分隔，但仅作为用户表达的
分隔喜好，整个二级标题只转为一个 message 元素。

在遇到另一个二级标题或一级标题时，结束这个对话段（message）。

## 忽略注释段

以 `# ` 开头的行，文档结构上像一级标题，语法上视为注释。
同样地，有单行注释与多行注释段。但按文档习惯，续行不必每行都加 `# ` 前缀。

二级标题但不满足可识别的对话段（`## role >>`），也忽略。一般可用于一级标题下面
的大段文档分隔。

## 引用块处理

在三反引号表示的引用块中，可能出现以 `# ` 或 `## ` 开始的行，但忽略其特殊语义。
例如：

```
引用块

## role >> 匹配二级标题，但不开启对话段，不转为 message
# 像注释
```

## 外部引入内容

- 用 `@` 引入文件内容，`!` 捕获命令输出
- 仅允许单行，标记 `@` 或 `!` 必须在行首，其后可留空格
- 但在三反引号的块中忽略 `@` 与 `!` 的特殊意义
- 在二级 `## role >>` 下面直接引入内容，插入当前 message

错误处理，如果 `@` 或 `!` 出错，输出原文本行，再于行尾隔一空格加上错误标记：
- 读取 `@` 文件失败或运行 `!` 命令失败，加 `(Read Error)` 后缀
- 当 `@` 或 `!` 无输出或仅空白输出时，加 `(Read Empty)` 后缀

# 对话示例

## system >> 请使用中文回答
## user >> 你好
## assistant >> 你好，大家好
## Q >> 1+2 是什么
## A >> 1+2=3
## Q >> 不是小学数学加法
是陈景润研究那个什么数论问题，1+2 是指什么，请科普一下。

# 转换结果

上节对话转为 json 片断的 message 数组形如：
```json
"messages": [
    {"role": "system", "content": "请使用中文回答"},
    {"role": "user", "content": "你好"},
    {"role": "assistant", "content": "你好，大家好"},
    {"role": "user", "content": "1+2 是什么"},
    {"role": "assistant", "content": "1+2=3"},
    {"role": "user", "content": "不是小学数学加法\n是陈景润研究那个什么数论问题，1+2 是指什么，请科普一下。"},
]
```
